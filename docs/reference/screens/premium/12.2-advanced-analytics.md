# 12.2. Advanced Workout Analytics (sub) — Premium

**Purpose:** The **advanced version of the "Basic Workout Analytics" card** on #12 History — **trends over time** plus a headline workload-balance estimate (ACWR), not a single-period snapshot. Where the Basic card compares *this* period to the *prior* one, this screen charts the **whole trajectory**: weekly training volume, HR efficiency, personal bests, training load, HR zones, and a now-anchored ACWR estimate. Premium-only; reached by tapping the **"Advanced ›"** action on the Basic Workout Analytics card (premium users only) — it's a drill-in from that card, not a separate menu entry.

*(Renamed from "Advanced Exercise Effect Estimates" — the heritage name leaned on the FYP brief's "estimate exercise effects" wording, but most of the screen's contents are stats/trends rather than estimates. Only the ACWR tile is genuinely an "estimate". The shorter "Workout Analytics" name is honest about what's on-screen and matches conventions in Strava / Garmin / Whoop.)*

This is the breadth feature that lifts Premium from "more detail" to "see your progress." Everything is **derived from existing `WorkoutSession` history** — no new entity required for v1.

## Layout

Vertical, scrollable. **No bottom nav** (sub-page). Back arrow → History (#12).

1. **Top bar** — ← back + `ADVANCED ANALYTICS` title.
2. **Workload ratio (ACWR)** — headline tile, **above** the range selector. ACWR is always now-anchored (last 7d vs last 28d), so placing it above the selector signals it's not scoped by the pills.
3. **Range selector** — pills: `4 wks` · `3 mo` · `1 yr` · `All` (Premium has no history cap, so the full range is available). Default is `4 wks` — short-term trend is the most common landing question. Month-based labels align with Strava / Garmin conventions; the four-pill set fits 340 px phone width without crowding. **Visually scopes only what's below it** — Workload ratio above and Personal bests at the bottom both ignore the selector by design.
4. **Trend sections** (each a titled card with a chart, range-scoped):
   - Weekly volume
   - HR efficiency
   - Training load
   - HR zones
5. **Personal bests** — all-time, range-independent. Bottom of the page.

## UI elements

### Range selector
Segmented pills controlling the window all charts aggregate over. Default `4 wks`. **Exception:** the ACWR tile is always anchored to "now" (last 7d vs last 28d) regardless of the range pill — ACWR is by definition a rolling-now metric, not a windowed average.

### Workload ratio (ACWR tile)
Sports-science **Acute:Chronic Workload Ratio** — a single number with a coloured band chip below it. Acute = sum of session loads over the last 7 days. Chronic = average weekly load over the last 28 days. Load per session = Training Effect score (1–10) from `lib/effectEstimate.computeTrainingEffect` — already per-user-calibrated via `FitnessProfile.restingHeartRate`.

| Ratio band | Label (chip colour) | Meaning |
|---|---|---|
| < 0.8 | **Detraining** (muted) | Load is dropping below baseline — fitness eroding |
| 0.8–1.3 | **Sustainable** (accent) | Sweet spot — progressive overload without overreach |
| 1.3–1.5 | **High Load** (gold) | Above baseline; getting stronger but watch fatigue |
| > 1.5 | **Overreaching** (danger) | Acute spike — injury / burnout risk |

Caption is descriptive ("Acute load ÷ chronic baseline. Sweet spot is 0.8–1.3."), never prescriptive — Honest AI rule. Bands are well-published (Gabbett et al.); the tile gives the user enough context to interpret the number themselves without telling them what to do.

When chronic baseline is too thin (< 0.5 average — roughly fewer than ~3 weeks of sessions), the tile shows "Not enough training history yet — needs ~3 weeks of sessions to compute" instead of a meaningless number.

### Weekly volume (line/bar)
Per-week totals over the range — **sessions**, **active minutes**, **calories** (toggle which metric the chart plots). Reuses `lib/periodAnalysis.ts` aggregation, extended to bucket by week across the range. Trend arrow vs the range's first half.

### HR efficiency (line)
For a chosen cardio type, average HR per week (or avg HR at a normalised pace) — a **downward trend at similar pace = improving aerobic fitness**. Derived from `WorkoutSession.avgHeartRate` grouped by week. Honest framing: label it "Avg HR per week (lower at the same pace = fitter)", not a fabricated "fitness score".

### Training load (bar)
Per-week sum of **Training Effect score** (`lib/effectEstimate.ts`) across sessions — shows ramp / taper. Flag big week-over-week jumps (>50%) as an "acute spike" caption (load-derived, factual — not advice).

### HR zones (stacked bar)
Karvonen **%HRR** breakdown of every HR sample across the selected range, binned into the five canonical zones:

| Zone | %HRR | Label | Colour |
|---|---|---|---|
| Z1 | 50–60% | Recovery | sky |
| Z2 | 60–70% | Aerobic | accent (lime) |
| Z3 | 70–80% | Tempo | yellow |
| Z4 | 80–90% | Threshold | orange |
| Z5 | 90–100% | VO2 max | red |

Cool→hot gradient signals intensity. Implementation walks each session's `trackPoints[]` HR samples per-second; sessions without tracking fall back to a single-zone allocation based on `avgHeartRate` × `durationSeconds`. Sub-Z1 time (below 50% HRR) is intentionally excluded — it's sedentary, not training.

Computed via `lib/advancedAnalytics.computeHrZones` using the user's `FitnessProfile.restingHeartRate` (default 60 bpm if null) and estimated max HR = `220 − age`. The footer cites both values so the user can sanity-check.

Caption is descriptive ("Z2 builds aerobic base; Z4 threshold; Z5 max effort.") — does not tell the user which zones they "should" spend more time in.

### Personal bests (stat tiles)
All-time bests from history: **longest distance**, **fastest pace** (cardio), **biggest session volume** (kg×reps, strength), **longest streak** (consecutive active days). Each tile shows the value + the date it was set.

## Edges

- **From:** History (#12) — the premium-only **"Advanced ›"** action in the header of the Basic Workout Analytics card.
- **To:** Back (←) → History (#12).

## Data touched

See [../../database-v1.md](../../database-v1.md).

- **Reads:** current user's `WorkoutSession` rows (all — Premium has no cap) + joined `WorkoutType`; `ExerciseLog` (for strength volume bests); `WorkoutSessionTrack` optional (for pace-normalised HR efficiency). Training Effect scores computed live per session.
- **Writes:** none — read-only analytics.

## Notes / non-obvious

- **One schema addition (v64).** `FitnessProfile.restingHeartRate` was added to back the Karvonen calculations on ACWR and HR Zones (also retro-improves Training Effect accuracy for users who set it). Null falls back to the population default 60 bpm.
- **No `MetricLog` entity yet.** Body-measurement trend toward a goal is still out of scope — `FitnessProfile.weightKg` is a single static field, not a time series. That needs a **`MetricLog`** entity (`userId`, `metric`, `value`, `recordedAt`); deferred. Advanced Analytics is *activity* trends, not *body-measurement* trends — copy should not imply the latter.
- **Honest framing throughout.** Charts describe what the numbers did (volume up, HR lower at pace, load spiked) — never motivational copy or "you should". Same rule as the Basic Workout Analytics card and Training Effect.
- **Relationship to Basic Workout Analytics (#12).** Basic = *this vs prior* period (already built, Premium gets the `All` pill). Advanced = *the full trend line* + ACWR estimate + HR zone breakdown. They're complementary; this screen is the deeper dive the History card links into.
- **Free has no entry point.** The "Advanced ›" action only renders for Premium on the Basic Workout Analytics card; Free sees that card's existing "Unlock with Premium" button instead. No locked teaser screen.
- **Charts reuse `components/LineChart.tsx`** where possible to stay consistent with #12.1's graphs.
