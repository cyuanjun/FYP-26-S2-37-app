# 12.1. History Detail (sub-page)

**Purpose:** Read-only recap of a specific past `WorkoutSession`. Same type-aware metrics + exercise log + feel/notes that the user saw on **#10 Workout Summary** when they finished, but viewed historically. Reached only by tapping a row on #12 History.

(Separate from #10 Summary because Summary is the "just finished" celebration with editable feel/notes and a Save & Finish CTA — Detail is a backwards-looking record with a Delete affordance.)

## Layout

Vertical, scrollable. **No bottom nav** (sub-page). Back arrow → History (#12).

Has a two-state shape: **read-only** (default) and **edit mode** (toggled from the top-right Edit button). Edit mode unlocks the fields the user can legitimately revise after the fact — **workout name, feel rating, notes, exercise logs, share toggle + description**. Everything else (date, duration, stats, training effect, graphs, planned-workout reference) is **dimmed to 50 % opacity** while editing so the user can see at a glance what's locked.

### Ownership gate
The Edit button only renders for the **session's owner** (`session.userId === current.userId`). Non-owners reach this screen only via the Social feed's embedded workout card — for them the page is purely read-only:

- **No Edit button** in the header (so no edit mode is reachable)
- **No Notes section** at all — `notes` is private to the owner; never shown to anyone else even if non-empty
- **No Delete button** (it's edit-mode-only anyway)
- **Description still renders** when `isShared = true` — it's the public caption, always visible

Stats, Training Effect, Graphs, and the Planned-workout reference remain visible for non-owners. These aren't marked private in the schema; they're the same data the Social feed card teases, just in fuller form.

### How editable fields are highlighted in edit mode
- **Section labels** for editable groups shift from `muted` to `accent` and gain a small ✏ **pencil glyph** before the text ("Workout name", "Exercises", "How it felt", "Notes")
- **Inputs and textareas** get a thicker accent-tinted border (`border-2 border-accent/40 focus:border-accent`) instead of the read-mode `border-faint`
- **Exercise card** gets an accent ring (`ring-2 ring-accent/30`) around the surface card so the whole list reads as "you can touch this"
- **Empty-state "+ Add" tile** uses accent ring + accent text to invite the first log

The combination of greying-out non-editables + accent-tinting editables makes the affordance unambiguous without adding noisy "EDIT" labels everywhere.

### Workout name (editable)
The h2 swaps for a full-width text input in edit mode, styled to roughly match the h2's visual weight (22 px font-black uppercase with the accent-tinted border). Placeholder shows the default fallback name so the user can see what they're overriding. Saves `customName` on each keystroke; empty input → `null` and the default cascade re-applies. A **"Reset to '{default}'"** text-link appears below the input only when `customName` is set — one-tap revert.

The name is *also* editable on #10 Summary at finish time (the natural moment, when context is fresh). This screen gives a second path for users who forgot or want to rename historical sessions later.

### Locked state (Free users only)
If a Free user deep-links to a session whose `EndedAt` is outside the current calendar month (older than the 1st of this month), the screen renders a **locked state** instead of the normal content:

- Back arrow + "WORKOUT" title (unchanged)
- Centred body: lock icon (🔒), "Outside your history window" heading, body copy explaining the cap, and a primary **⚡ Upgrade to Premium** CTA → #16

The session row is filtered out of the #12 list for Free users too, so this is a defence-against-direct-navigation path rather than a normal flow. Premium users never see this state — they have unlimited history.

## UI elements

### Top bar
- *Left:* ← back arrow + "WORKOUT" label. In edit mode a small **EDITING** pill in `accent` appears next to the label (tinted bg, accent ring) — gives the user an unmistakable cue about the current mode beyond just the button state.
- *Right:* **Edit** button (accent text-link). In edit mode it becomes a filled accent **Done** pill that exits back to read-only.

### Session header
- **Workout name** (display 26 px font-black uppercase tracking-tight) — falls back through `WorkoutSession.customName ?? PlannedWorkout.name ?? WorkoutType.name ?? "Workout"`. Editable in edit mode (see "Workout name" section above).
- Meta line: relative date + duration in muted Caption 2 (e.g. "Tue 20 May · 33 min"). Appends `· Freeform` in `accent` when `PlannedWorkoutID IS NULL`. **Dims to 50 % opacity in edit mode** — these are recorded facts and never editable.

### Stats grid
**Same type-aware logic as #10:**
- Cardio: Duration / Distance / Avg Pace / Calories / Avg HR / Max HR (2 × 3)
- Non-cardio: Duration / Calories / Avg HR / Max HR (2 × 2)

### Training Effect (Free)
Same shared `TrainingEffectCard` used on #10 Summary — slotted between the Stats grid and the Graphs section. See [10 Summary → Training Effect](10-workout-summary.md#training-effect-free) for the full spec, compute formula, and Premium expansion plan. The card behaves identically here; History is just the historical viewing surface.

*(The old "AI Session Summary" card was removed from both #10 and #12.1 — it only restated the stat tiles. Zone time / pace splits / cadence live in the Graphs below; cross-session trends live on #12.2 Advanced Workout Analytics.)*

### Graphs (any session with a recorded track)
Below the stats grid for any session that has a linked `WorkoutSessionTrack`. Cardio gets the full graph experience; non-cardio gets a simpler HR-only chart.

**Cardio sessions** (`running`, `cycling`, `swimming`, `hiit`):
- Section header: `GRAPHS` (left) + a small **source label** (right) — resolves to the actual device name when possible: `From {ConnectedDevice.deviceName}` (e.g. "From Mia's Apple Watch"); falls back to `From phone sensors` when `DataSource = phone_sensors` with no device; `From manual entry` when there's no device or sensor at all; `From GPX import` when `WorkoutSessionTrack.source = 'gpx'`. Generic "wearable" is intentionally avoided — showing the device name validates which gadget supplied the data.
- 4-pill toggle row: **HR / Pace / Cadence / Elev** (active pill = filled accent; others = surface chip)
- Chart card (see below)

**Non-cardio sessions** (`strength`, `yoga`, `pilates`, `boxing`, `climbing`, etc.):
- Section header swaps to `HEART RATE` (left) + same source label (right)
- **No pill row** — there's only one metric to show (HR), so the toggle would be a single useless pill. Pace / cadence / elevation don't apply on a yoga mat or bench.
- Chart card renders directly under the header, locked to the HR series

**Chart card** (shared by both):
- Metric name + unit row (e.g. `Heart Rate · bpm` — cadence unit auto-swaps to `rpm` when the workout type is cycling)
- `LineChart` component — auto-scales y-axis to data min/max, draws a stroked path with a 12 %-opacity area fill in the metric's colour (HR red, Pace lime, Cadence gold, Elev info-blue)
- X-axis hint row: `0:00` → `{total duration}` (no tick marks — phone width doesn't have room)
- **Avg / Max-or-Fastest / Min-or-Slowest** stats row below, separated by a faint divider. For Pace, the "best" tile is labelled `Fastest` and shows the *minimum* sec/km value (lower is better); HR/Cadence/Elev label them `Max` and `Min`
- Empty metric state: if a metric has no readings (e.g. an indoor cycling session with no GPS elevation), the chart card shows `Not captured for this session.` instead of broken numbers

Tap a pill (cardio only) → swaps the chart's data series in place. No new screen, no modal.

### Planned Workout card (plan-based only)
Reference card surfacing what the plan actually called for. Helps the user compare planned vs actual (e.g. planned 45 min → actual 32 min). Only renders when `PlannedWorkoutID` is set.

- `FROM THE PLAN` eyebrow
- Planned workout name (display 18 px font-black uppercase)
- Type · target duration · week (Caption 2 muted uppercase)
- Plan name (muted Footnote)
- Planned workout descriptor (Subheadline ink) — the coaching detail (e.g. "Squat, press, row — 4 sets of 8 reps each")

### Exercises (non-cardio only)
**Read-only mode:** rows display exercise name + `Sets × Reps · Weight`; tapping does nothing.

**Edit mode:** rows become tap targets that open the shared `ExerciseLogModal` (same component used on #10 Summary) in edit-with-delete. A `+ Add exercise` row appears at the bottom of the card. When there are zero exercise logs and the user enters edit mode, the empty-state "+ Add what you did" tile appears (same affordance as the post-session empty state on #10).

### Feel rating
**Read-only:** single accent chip showing the captured `FeelRating`. Hidden if `FeelRating IS NULL`.

**Edit mode:** 4-column chip grid (same component as #10) — tap to set, tap again to clear. Writes immediately to `WorkoutSession.FeelRating`.

### Share to Social + Description
"Is this session shared?" derives from `posts.find(p => p.kind === 'workout_share' && p.workoutSessionId === session.workoutSessionId)`.

**Read-only:** when a workout-share Post exists, renders a `DESCRIPTION` section showing `Post.body` (or "Shared without a description." when body is null). Hidden entirely when the session has no Post.

**Edit mode:** renders a `SHARE TO SOCIAL` section with an iOS-style toggle. Toggling ON calls `createWorkoutSharePost({...})`; toggling OFF calls `deletePost(postId)`. When a Post exists, a textarea for the public caption appears below the toggle and writes to `Post.body`. Toggling off pulls the session from the feed and removes its likes/comments (cascade).

### Notes (private)
**Read-only:** paragraph display, hidden if `Notes IS NULL`. Section label reads `NOTES (private)` to make the privacy contract obvious — these are owner-only, never shared even when the session is.

**Edit mode:** editable textarea — writes immediately to `WorkoutSession.notes` on each keystroke (blank input becomes `null`). Same `(private)` label on the section header.

### "View on Social" link (owner + shared sessions only)
When a workout-share `Post` exists for **the owner's** session, render a compact accent-tinted link card at the bottom of the read content: **💬 View comments & likes on Social →**, routing to #11.1 Post Detail.

**Owner-only on purpose.** Non-owners always reach #12.1 via the embedded workout card on #11.1 Post Detail — they're already in the social context, so a back-link to where they came from is a confusing loop. Owners, on the other hand, arrive at #12.1 from their own History list with no social context; the link card is what surfaces "this is shared and people are interacting with it."

Comments + likes no longer live on this page. Earlier iterations had a full Discussion section here (likes / comments thread / reply input) but the model shifted: comments are a *social* interaction, they belong with the Post on a social surface (#11.1), not on the workout-data page. #12.1 stays focused on workout data; #11.1 owns the social side. The link card is the bridge — and only the owner needs that bridge.

The link card is hidden when no Post exists for the session (i.e. the session was never shared) OR when the viewer isn't the owner.

### Delete action
Full-width **outlined danger button** at the bottom of **edit mode only** — `border-danger/40 text-danger` with `bg-danger/10` on hover, font-black uppercase tracking-wider to match other primary actions. Lives with the other destructive edit-mode controls; the read-only view is purely for review and never offers Delete (reduces accidental destruction during casual scrolling, and groups "operations that change/remove data" under one mode).

Tap → confirmation modal:
- Title: "Delete this session?"
- Body: "Stats, exercises, and notes will be removed. This can't be undone."
- Primary: "Delete" (filled danger)
- Secondary: "Cancel"

On confirm: deletes the `WorkoutSession` row + cascades to `ExerciseLog` rows, then navigates back to #12 History.

## Edges

- **From:**
  - History (#12) — row tap (owner)
  - Post Detail (#11.1) — embedded workout-card tap (owner OR non-owner). The link carries a `?fromPostId=…` query param so the back arrow returns to the post instead of dumping the viewer onto a History list they didn't come from.
- **To:**
  - Back (←) → Post Detail (#11.1) when `fromPostId` is present, otherwise History (#12). Label flips between "Back to Post" / "Back to History".
  - History (#12) — on Delete confirm (always; the post-detail back target is moot after the session is gone)

## Data touched

See [../../database-v1.md](../../database-v1.md).

- **Reads:** the specific `WorkoutSession` (by `WorkoutSessionID` route param); joined `WorkoutType` + `PlannedWorkout` + `FitnessPlan`; all `ExerciseLog` rows for the session; the linked `WorkoutSessionTrack` if one exists (cardio sessions only — drives the graphs section); the `Post` row of kind `workout_share` for this session if one exists (drives Description + the "View on Social" link)
- **Writes (edit mode):**
  - `WorkoutSession.CustomName` — on title-input change (null when empty so the default cascade re-applies)
  - `WorkoutSession.FeelRating` — on chip tap (set or clear)
  - `WorkoutSession.Notes` — on textarea change (private)
  - `Post` row create/delete via `createWorkoutSharePost` / `deletePost` — on Share toggle
  - `Post.Body` — on description textarea change (public caption when shared)
  - `ExerciseLog` rows — insert / update / delete via the shared `ExerciseLogModal`
  - **Not editable here:** date, duration, stats, track data, planned-workout reference, Training Effect — dimmed to 50 % opacity to telegraph it.
- **Writes (on Delete):** removes the `WorkoutSession` row and all linked `ExerciseLog` rows. No soft-delete in v1 — the user explicitly confirmed.

## Notes / non-obvious

- **Edit mode is opinion + log only.** We let the user revise their feel rating, notes, and exercise list because those are user-entered. Stats from sensors stay locked — editing duration / distance / HR would erode trust in history. The Done button is just a UI state toggle (every change writes immediately) — there's no "Cancel" because there's nothing to cancel against.
- **Plan progress isn't recomputed on delete.** Deleting a historical session doesn't roll back a plan's completion count beyond what the user sees — the count is derived anyway, so the next render reflects the new total automatically.
