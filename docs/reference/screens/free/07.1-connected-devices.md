# 7.1. Connected Devices (sub of #7)

**Purpose:** Manage paired wearables and sensors used as data sources during workout sessions. Reached from #7 Train's "Connected Devices" status row.

**Sub-numbering convention:** `7.x` = sub-screen reached from #7's UI (same pattern as `13.1` Fitness Profile under Profile).

## Layout

Vertical, scrollable. **No bottom nav** (sub-page). Back arrow → Train (#7).

## UI elements

### Top bar
- ← back arrow + "CONNECTED DEVICES" in **Title 1**-ish (26 px `ink` font-black uppercase tracking-tight)

### Paired devices list (or empty state)
If 1+ devices: a `divide-y divide-faint` list of device rows.

Each row:
- *Left:* small device icon (Apple Watch / Fitbit / etc.) + device name + connection status pill
- *Right:* chevron → opens per-device sub-page (TBD — rename, sync settings, remove)
- *Below name (muted, Subheadline):* "Last synced: 2 min ago" or "Not connected"

Empty state: centred body text "No devices paired yet" + small description.

### Add Device CTA
Full-width outlined-accent button at the bottom (above safe area):
- "+ Add device" → opens a mock Bluetooth scan modal (TBD — for FYP, can be a fake list of discoverable devices to pick from)

## Edges

- **From:** Train (#7) — tap Connected Devices status row
- **To:**
  - Back (←) → Train (#7) — hard-wired `Link`
  - Per-device detail/edit sub-page (TBD)
  - "Add device" scan modal (TBD — uses `Modal` component, lists fake discoverable devices to pick)

## Data touched

See [../../database-v1.md](../../database-v1.md).

- **Reads:** all `ConnectedDevice` rows for current user
- **Writes:** add — inserts a new `ConnectedDevice`; rename — updates `DeviceName`; toggle active — flips `IsActive`; remove — deletes the row; sync — updates `LastSyncedAt`

## Notes / non-obvious

- **Mock pairing.** For the FYP scope, the "Bluetooth scan" is a fake modal listing common device types. In a real build this would invoke `navigator.bluetooth` or native plugins.
- **Phone sensors as a virtual device.** Even without a paired wearable, `phone_sensors` is always available — pre-seeded as a system-managed `ConnectedDevice` with `DeviceType = 'phone_sensors'`. Always shown at the top of the list, can't be removed.
- **Why FK to User (not UserFitnessProfile)?** Devices are user-account-level. If we later add Admin/Expert flows that need device data (e.g. expert syncs their watch for their own training), they'd want a `ConnectedDevice` without needing a `UserFitnessProfile`. Cheap future-proofing.
