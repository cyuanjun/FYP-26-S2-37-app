# 13.3. Account Settings

**Purpose:** Account-level data the user can edit — identity, contact, app preference, and credential management. Reached from Profile (#13) → "Account Settings" menu row.

**Scope is deliberately tight:** anything **fitness-related** (DOB, Sex, Height, Weight, Goal) lives on the future **Fitness Profile** sub-page (#13.x), not here. This screen is for *who you are as an account*, not *who you are as an athlete*.

## Layout

Vertical, scrollable. **No bottom nav** — sub-page accessed from Profile; back arrow returns.

iOS grouped-settings pattern: section label in `muted` uppercase, then a column of `SettingRow`s separated by `faint` dividers.

## UI elements

### Top bar
- Left-arrow icon (22 px) + "ACCOUNT SETTINGS" title in **Title 1**-ish (26 px `ink` font-black uppercase tracking-tight)
- No right action

### PERSONAL INFO section
Section label in **Caption 2** (11 px `muted` uppercase tracking 0.18em font-semibold), then four `SettingRow`s — each: label in **Body** (17 px `ink` font-semibold) above value in **Subheadline** (15 px `muted`), with a chevron-right in `muted` indicating tap-to-edit.

- Full Name — composed from `User.FirstName` + `User.LastName`
- Username — `User.Username` ("@miapatel")
- Email — `User.Email`
- Phone — `User.Phone` (new)

Each row taps through to a per-field edit screen (future sub-pages, not built yet).

### PREFERENCES section
- One row: "Preferred Measurement Unit" label with a `Segmented` control on the right — METRIC / IMPERIAL toggle. Active option fills `accent`, inactive is `muted`. Persists to `User.PreferredUnits` instantly on change.

### SECURITY section
- Single full-width outlined button "CHANGE PASSWORD" — 1 px `faint` border, `ink` text, **Body** (17 px) font-black uppercase tracking-wider, `rounded-2xl`, transparent bg, hover sharpens border to `muted`. **Neutral styling** — Change Password is a routine action, not destructive (Delete Account would be the danger-coloured one).

Colours from [../../palette.md](../../palette.md), type sizes from [../../typography.md](../../typography.md). Reuses: `SettingRow` and `Segmented` (new); inline section label helper.

## Edges

- **From:** Profile (#13) — tap "Account Settings" menu row
- **To:**
  - Back (←) → Profile (#13) — hard-wired `Link` (mirrors Profile's pattern; revisit when more entry points exist)
  - Per-field edit screens (TBD sub-pages: Edit Full Name, Edit Username, Edit Email, Edit Phone) — not built yet; tap currently a no-op
  - Change Password → likely opens a sub-page that re-uses the Forgot password reset-link flow

## Data touched

See [../../database-v1.md](../../database-v1.md).

- **Reads:** `User.FirstName`, `User.LastName`, `User.Username`, `User.Email`, `User.Phone` (new), `User.PreferredUnits` (new)
- **Writes:** `User.PreferredUnits` (on Segmented change — instant, no save button)
- **Triggers:** Change Password → server flow that emails a reset link (same machinery as #4 Forgot password)

## Notes / non-obvious

- **DOB and Sex are deliberately not here.** They moved to the future Fitness Profile sub-page because in this app they're fitness inputs (BMR, calorie estimate, HR zones), not account-management inputs. Grouping by *use* over by *type of data*.
- **No "Save" buttons.** Inline edits commit on row sub-page submit; the segmented control commits instantly. Standard iOS settings UX.
- **No Delete Account button.** Deferred — when it lands, it gets the only `danger`-filled button on this screen. Change Password stays neutral so the visual hierarchy reads clearly: outlined-neutral = routine, filled-danger = irreversible.
