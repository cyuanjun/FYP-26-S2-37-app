# 13.5. Submit Feedback (sub)

**Purpose:** Lightweight, fire-and-forget channel for users to report bugs, suggest features, or share general thoughts. Reached only from **Profile** (#13) → "Submit Feedback" menu row. The mock has no inbound surface (no inbox, no triage, no reply) — submissions land in the `Feedback` table and that's the end of the user-facing flow.

## Layout

Vertical, no scroll on idle (form fits the phone frame). **No bottom nav** (sub-page). Pinned footer for the Submit CTA so it stays reachable while typing on a soft keyboard. Back arrow → Profile (#13).

## UI elements

### Top bar
← back arrow + `SUBMIT FEEDBACK` title in caption-bold uppercase tracking `[0.14em]`.

### Intro line
Muted Subheadline directly under the header: *"Found a bug or have an idea for the app? Tell us — we read every submission."*

### Category — segmented tiles (single-select)
Three full-width tiles, one per category. Stacked vertically with `gap-1.5` so the active tile is obvious without a header. Each tile shows label (Caption 1 black uppercase) + one-line hint (Caption 2 muted).

| Value | Label | Hint |
|---|---|---|
| `bug` | Bug | Something is broken or behaving unexpectedly |
| `feature_request` | Feature request | Idea for something new you'd like to see |
| `general` | General | Praise, complaints, or anything else |

Active state: `bg-accent/10` background + `ring-accent` outline + `text-accent` label. Inactive: `bg-surface` + `ring-faint/60` + `text-ink`. Default selection is `general` so a user who taps Submit without thinking still has a valid value.

### Body — textarea
`rows={8}` textarea, `bg-surface` with `border-faint` → focus border `accent`. Placeholder is **category-aware**:

- `bug` → *"What happened? What did you expect?"*
- `feature_request` → *"What would you like to see? How would it help you?"*
- `general` → *"Tell us what's on your mind…"*

Below the textarea, a Caption-2 counter shows either *"N more character(s) needed"* (red-ish via `text-faint`) until the trimmed body hits the **10-character minimum**, then flips to *"N characters"* once valid.

### Pinned footer — Submit CTA
Full-width accent button `Submit feedback`, sticky at the bottom inside a `border-t border-faint/40` divider. **Disabled** (faint ring) until `trimmedBody.length >= 10`. On tap: writes a new `Feedback` row via `submitFeedback({ userId, category, body })` and swaps the screen body into the success state below.

### Success state (replaces the form)
After a successful submit, the form is unmounted and replaced with a centered success view inside the same screen (back arrow still works, header still says SUBMIT FEEDBACK):

- Large accent check-circle glyph
- Title (Title 2, black uppercase): *"Thanks for your feedback"*
- Muted body line: *"We read every submission. Bugs go to the team for triage; ideas help shape what we build next."*
- Two stacked buttons:
  - **Submit another** (outline) — clears state and returns to the empty form
  - **Back to Profile** (accent) — routes to `/free/13-profile`

## Edges

- **From:** Profile (#13) — "Submit Feedback" menu row (only entry point)
- **To:**
  - Back (←) → Profile (#13)
  - "Back to Profile" button in success state → Profile (#13)
  - "Submit another" button in success state → back to empty form (in-screen)

## Data touched

See [../../database-v1.md](../../database-v1.md).

- **Reads:** `User.UserID` (attached to the submitted feedback row as the FK)
- **Writes (on Submit):** appends a new `Feedback` row via `submitFeedback({ userId, category, body })` in `db.ts`:

  | Attribute | Type | Notes |
  |---|---|---|
  | FeedbackID | UUID | PK (`fb_<ts>_<rand>`) |
  | UserID | UUID | FK → User (the submitter) |
  | Category | enum | `bug` \| `feature_request` \| `general` |
  | Body | text | required, non-empty after trim, ≥10 chars |
  | CreatedAt | timestamp | wall-clock at submit time |

## Notes / non-obvious

- **Fire-and-forget on purpose.** There is no acknowledgement, triage, response, or read-receipt state on the `Feedback` row because there is no internal-facing surface in the mock to consume it. If/when a support inbox is built (admin role), add `Status` + `AssigneeID` + `ResolvedAt` columns rather than a parallel entity.
- **10-character minimum, not 0.** A 1-character body ("k") is technically non-empty but useless for triage. 10 chars is enough to force at least a short sentence without being annoying. Counted *after* `trim()` so leading whitespace doesn't pad the minimum.
- **Default category is `general`, not unselected.** Forcing the user to actively pick a category before they can even type would be friction without payoff — the category is a hint for routing, not a hard constraint. A user who taps Submit on autopilot still produces a valid row.
- **Placeholder copy switches by category.** Three different prompts because "what happened" makes no sense for praise and "tell us what's on your mind" makes no sense when triaging a bug. Cheap UX win.
- **Success state stays on the screen.** Routing back to Profile + showing a toast was the obvious alternative; chose in-screen success because it makes "Submit another" a one-tap action without a round-trip, and the explicit "Back to Profile" button still gets users out in one tap.
- **No spam handling, no rate limit.** Real backend will need both (per-user submit cap, basic profanity/length filter). Out of scope for a UI mock with no server.
- **No `AppVersion` / device metadata.** A real client would attach these for triage; deferred until there's an actual triage surface that uses them. Adding the column now without a consumer is YAGNI.
- **Premium delta — none planned.** Feedback is universally available; gating it would be hostile. If Premium ever gets a faster-response SLA, that's a server-side promise, not a UI feature.
