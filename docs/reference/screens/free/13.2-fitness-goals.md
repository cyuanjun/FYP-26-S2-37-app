# 13.2. Fitness Goals (sub of #13)

**Purpose:** Capture what the athlete is trying to achieve so the app can recommend workouts, surface progress, and motivate via reminders. Reached from Profile (#13) → "Fitness Goals" menu row.

## Layout

Vertical, scrollable. No bottom nav. Back arrow → Profile (#13).

## UI elements

### Top bar
- ← back arrow + "FITNESS GOALS" in **Title 1**-ish (26 px `ink` font-black uppercase tracking-tight)

### PRIMARY GOAL section
Vertically-stacked rich cards, single-select (one always active). Each card has:
- Title in **Body** (17 px `ink` font-bold) — e.g. "Lose Weight"
- Descriptor in **Footnote** (13 px `muted`) — e.g. "Calorie deficit · cardio focus"
- Rounded-2xl `surface` background with 1 px `faint` border
- Selected state: `accent` border + 1 px accent ring + 28 px `accent` circle with `bg`-coloured checkmark, vertically centred on the right edge

Options:
- **Lose Weight** — "Calorie deficit · cardio focus"
- **Build Muscle** — "Strength training · progressive overload"
- **Improve Endurance** — "Cardio · long sessions"
- **Maintain Fitness** — "Balanced routine"

### GOAL TARGET section (conditional)
Hidden when `PrimaryGoal === 'maintain_fitness'` (maintaining doesn't have a target).

Single full-width `surface` card containing a horizontal stepper:
- 48 × 48 `surface-2` − button (left)
- Centred large number + uppercase unit ("57 KG", "60 MINS") in display 38 px font-black tracking-tight
- Subtitle below the number in **Caption 1** (12 px `muted`) — context-dependent:
  - `lose_weight` → "−5 kg from current weight" (delta from `UserFitnessProfile.weightKg`)
  - `build_muscle` → "+4 kg lean mass" (delta from current weight)
  - `improve_endurance` → "Sustained activity" (no delta)
- 48 × 48 `surface-2` + button (right)

**Step amounts** per goal type:
- `kg` units: ±1
- `minutes` units: ±5

### WEEKLY COMMITMENT section
**Stepper** (same component as Goal Target — `Stepper` pattern reused for consistency):
- Display: large number + uppercase "DAYS" label, with "per week" subtitle below
- Bounded 1–7 (decrement clamped at 1, increment clamped at 7)
- Shown for all primary goals — including `maintain_fitness`, where frequency is the only commitment that still matters

### TIMELINE (WEEKS) section (conditional)
Hidden when `PrimaryGoal === 'maintain_fitness'` — maintenance is an ongoing routine, not a duration-bounded goal. Save writes `TimelineWeeks: null` for maintenance.

For other goals: row of round chips (44 × 44 px, `rounded-full`). Single-select; one always active.
- Options: **4 · 8 · 12 · 16 · 24** (locked set)
- Active fills `accent`, inactive is outlined `muted`/`faint`

Relative-duration model replaces an absolute deadline date — easier to set, more relatable.

**Section order rationale:** Weekly Commitment goes above Timeline because frequency is universal (every goal has one) while duration is conditional. Putting the universal element first means the screen layout doesn't shift around when switching to Maintain Fitness — Goal Target disappears, then Weekly Commitment stays put, Timeline drops off the bottom.

### SAVE GOAL button
Full-width primary CTA at the bottom. **Body** (17 px) font-black uppercase tracking-wider, `accent` bg, `bg` text, accent glow shadow. On tap → `upsertCurrentFitnessGoal(userId, patch)`. Store finds the active goal (`AchievedAt IS NULL`) for the user and patches it, or inserts a new one with sensible defaults if none exists.

Colours from [../../palette.md](../../palette.md), type sizes from [../../typography.md](../../typography.md). Reuses: `SectionLabel`. Inline component definitions for `GoalCard`, `Stepper`, `RoundChip` (extract to shared components if reused elsewhere).

## Edges

- **From:** Profile (#13) — tap "Fitness Goals" menu row
- **To:**
  - Back (←) → Profile (#13) — hard-wired `Link`
  - Per-field edit sub-pages (TBD: Edit Target Weight, Edit Daily Steps, Edit Weekly Days, Edit Deadline)
  - On Save → stays on screen (or returns to Profile with success toast — TBD)

## Data touched

See [../../database-v1.md](../../database-v1.md).

- **Reads:** `User.userId`; `UserFitnessProfile.weightKg` (used as `StartingValue` for weight-based goals); active `FitnessGoal` for the user (`AchievedAt IS NULL`)
- **Writes (on Save):** upserts the active `FitnessGoal` row with: `PrimaryGoal`, `TargetValue`, `TargetUnit`, `StartingValue`, `TimelineWeeks`, `WeeklyCommitmentDays`

## Notes / non-obvious

- **One active goal at a time.** Convention: the user's active goal is the row with `achievedAt IS NULL`. Achieving a goal sets `AchievedAt`; setting a new goal creates a new row with `AchievedAt = null`. Historical goals remain in the table for trend / achievement tracking later.
- **Polymorphic target.** `TargetValue` + `TargetUnit` together describe the goal. The screen's stepper changes increment + unit label based on `PrimaryGoal`; the DB doesn't enforce the link (any enum value is allowed with any unit) — application logic owns the mapping.
- **`maintain_fitness` has no target.** Both `TargetValue` and `TargetUnit` write as `null`. The screen hides the Goal Target section entirely when this goal is selected.
- **Goal change resets target.** When the user switches `PrimaryGoal` in the screen, `TargetValue` resets to that goal's default (e.g. 57 kg for lose weight, 60 mins for endurance). Keeping a stale value across a unit change ("57" carried over from kg to minutes) would be misleading.
- **`StartingValue` is auto-captured** from `UserFitnessProfile.weightKg` for weight goals at the moment the goal is saved. Lets the screen show "-5 kg from current weight" without re-querying historical weight logs.
- **Save behaviour is upsert.** The store's `upsertCurrentFitnessGoal(userId, patch)` finds the active goal or creates one — the screen never needs to know whether it's editing or creating.
