# 11.1. Post Detail (sub)

**Purpose:** The single page where any `Post` lives in full — wrapped content, like toggle, comments thread, reply input. **Polymorphic by post kind** (works for `workout_share` and `challenge_result` today; future post kinds slot in here with no rewrites to likes / comments plumbing). Reached from any post card on the Social feed.

(This screen was prototyped, deferred during the workouts-are-posts simplification, then brought back when the polymorphic Post model landed. The current architecture earns its keep — Social interactions stay on social surfaces, workout-data stays on #12.1 History Detail.)

## Why comments don't live on #12.1 anymore

The earlier-built Discussion section on #12.1 mixed two unrelated concerns: workout *data* (stats, graphs, exercises) and social *interactions* (comments, likes). With polymorphic posts that include non-workout content (challenge results), bundling comments into the workout-detail page no longer made sense — challenge-result posts have no workout to attach comments to. Comments belong with the Post, not with whatever the Post wraps. So #12.1 stays focused on workout data; #11.1 owns the social side. The owner sees a small **💬 View comments & likes on Social →** card on their #12.1 page for shared sessions, which routes here.

## Layout

Vertical, scrollable. **No bottom nav** (sub-page). Back arrow → Social (#11).

## UI elements

### Top bar
← back arrow + `POST` label in caption-bold.

### Author row
40 px circular avatar (initial fallback) + author name (Subheadline ink font-bold) + `@handle · relative time` (Caption 2 muted). **For the post's author**, a 36 px circular pencil button sits on the right of the row — single entry-point for editing the caption inline (whether the body exists or is empty). Non-authors see no button.

### Optional caption (author-editable via the author-row pencil)
If `Post.body` is non-empty, render as a paragraph in `ink` Subheadline below the author row. If empty, render nothing — the empty case is signalled only by the pencil button's `aria-label` flipping from *Edit description* to *Add description*. Tapping the pencil swaps the caption block for an accent-bordered textarea + **Cancel** / **Save** pills (right-aligned). Save writes through `updatePostBody` (trimmed empty input → `null`); Cancel discards the draft.

This is a parallel edit surface to the Description textarea on #12.1 History Detail's edit mode — both write the same `Post.body` field. The duplication is intentional: #12.1 owns retroactive *workout* edits and bundles the description with the share toggle there; #11.1 lets the author fix typos / amend the caption without bouncing through the workout's edit mode, which is the natural impulse after a comment thread starts. Non-authors see only the static paragraph (or nothing).

### Polymorphic content body

**`kind = 'workout_share'`** — renders the shared **`WorkoutListCard`** (same component used on #12 History list + Social feed). Top row + meta + 3-metric strip. Tap the card → #12.1 History Detail for that session, for users who want the full workout breakdown.

**`kind = 'challenge_result'`** — renders the same accent-tinted result panel as on the Social feed:
- `🏆 CHALLENGE RESULT` eyebrow + challenge name + metric label
- Optional challenge description
- Ranked list (🥇 🥈 🥉 for top 3, `N.` for 4+, `—` for unsubmitted) with metric value per row

Future post kinds (PR celebration, badge earned, etc.) plug in here as new branches of the same renderer.

### Like + count + share strip
Below the content body, a thin strip:
- ❤ Like toggle (heart fills `danger` red when liked, `muted` otherwise; count shown when > 0)
- "N comments" / "No comments yet" caption
- ↑ Share button — opens the **Share post modal** (see [#11 Social spec → "Share post modal"](11-social.md#share-post-modal)) with X / Facebook / Instagram / TikTok / Copy link options. Fire-and-forget; not tracked in the schema.

### Comments thread
Each comment is its own `surface` card with:
- 32 px avatar + author name + relative time
- Body paragraph
- Current-user comments get a small `×` delete affordance on the right

Sort by `CreatedAt ASC` (oldest first — natural conversation order).

### Reply input (always at the bottom of the scroll)
Surface card with a single-row textarea + `Post` pill button. Button disabled until the trimmed draft is non-empty.

## Edges

- **From:** Social (#11) — tap any post card; the same routing applies for both workout-share and challenge-result posts
- **From:** History Detail (#12.1) — owner's `💬 View comments & likes on Social →` link for shared sessions
- **To:**
  - Back (←) → Social (#11)
  - History Detail (#12.1) → embedded workout-card tap on a workout-share post
  - User Profile (#11.2) → tap the post author's avatar/name, a challenge ranking row's name, or a comment's avatar/name

## Data touched

See [../../database-v1.md](../../database-v1.md).

- **Reads:** the specific `Post` (by `PostID` route param), its author from `findUserById`, all `PostLike` + `PostComment` rows for the post, comment authors via `findUserById`; for `workout_share` posts: the linked `WorkoutSession` + `WorkoutType` + `ExerciseLog` rows; for `challenge_result` posts: the linked `Challenge` + its `ChallengeParticipant` rows + each participant's submitted (or best in-window) `WorkoutSession` (to compute rankings)
- **Writes:**
  - `togglePostLike(postId, currentUserId)` — heart toggle
  - `addPostComment({ postId, userId, body })` — reply input's Post button
  - `deletePostComment(commentId)` — own comment's `×` button
  - `updatePostBody(postId, body)` — inline caption editor (author-only); trimmed empty input writes `null`

## Notes / non-obvious

- **One detail page, any post kind.** Adding a new `PostKind` later means: extend the polymorphic body switch, add a new wrap-data lookup, done. Comments + likes are untouched.
- **No deletion of posts from here.** Workout-share posts get un-shared by toggling Share off on #10 Summary or #12.1 edit mode (which calls `deletePost`). Challenge-result posts can't be deleted in v1 — they're system-generated when a `best_of` challenge's deadline passes.
- **Embedded workout card on share posts navigates to #12.1.** A user who lands on Post Detail and wants to see the full stats grid, graphs, training effect, etc. taps the workout card to jump to History Detail. The two pages compose cleanly without overlapping responsibility.
- **Shared ranking helper.** `rankChallengeParticipants` lives in `lib/challengeProgress.ts` and is imported by every surface that ranks Challenge participants (Social Challenges tab, this Post Detail page, #11.3 Challenge Detail). It branches on `metricKind`: for `accumulator` it sums each participant's qualifying sessions live; for `best_of` it auto-picks each participant's best qualifying session in the window when no explicit `workoutSessionId` is set — so leaderboards aren't perma-hidden waiting for users to "submit". For ended best_of challenges, the explicit submission is the final entry.
