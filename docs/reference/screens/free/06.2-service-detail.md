# 6.2. Service Detail (sub)

**Purpose:** Full detail page for a single `ExpertService` — what's included, who offers it, and a pinned-footer **Request** CTA (no payment — the client sends a request with a goal message; the expert accepts/declines on #22). Once accepted, the expert's `Deliverable` documents show here, and the client can leave a review. Reached from either #6's Service Listings sub-tab or #6.1's "Service listings" section on an expert's profile.

## Layout

Vertical, scrollable, with a **pinned footer CTA** (same pattern as #11.3 Challenge Detail's Join button). Back arrow uses `navigate(-1)` to preserve wherever the user came from (#6 sub-tab + scroll, OR #6.1 if they tapped a service from a profile).

## UI elements

### Top bar
← back arrow + `SERVICE` label in caption-bold.

### Hero
- Caption-2 muted: category label (e.g. *STRENGTH*)
- Display 22 px font-black uppercase service name
- 14 px muted description paragraph
- Display 28 px font-black price (e.g. `$120`)

### From {expert} — deliverables (only after acceptance)
If the client has an accepted (`completed`) request for this service, a `FROM {FIRSTNAME}` section lists the `Deliverable` documents the expert has sent (newest first). Each is a card: title + relative time + optional note, then the `sections → items` (heading + `label · detail · sub` rows). Hidden entirely when there's no accepted engagement.

### What's included
Section header `WHAT'S INCLUDED` + a `surface` card. Each `service.detailBullets[]` entry renders as a row: accent check glyph + the bullet text in ink 14 px.

### Offered by
Section header `OFFERED BY` + a `surface` card with the expert's mini profile:
- 48 px circular avatar (initials)
- Name + title
- Rating `★ X.X` in accent (right-aligned)
- Footer caption: `View full profile ›`

Whole card is a `Link` to **#6.1 Expert Detail** for the offering expert.

### Pinned footer — state machine
A footer band below the scrollable content, separated by `border-t border-faint/40`. The single pinned action reflects the client's `ServiceRequest` for this service (cancelled/declined ones don't count):

- **No request yet** → full-width accent CTA `Request · $XXX`. Tap opens the **Request modal**.
- **`pending`** → gold caption card `Request pending` (awaiting the expert's accept/decline on #22). No action.
- **`completed` (accepted), not yet reviewed** → accent CTA `Leave a review`. Tap opens the **Review modal**.
- **`completed`, already reviewed** → accent-tinted caption card `✓ Reviewed`.

### Request modal
Standard `Modal` triggered by the Request CTA. Contents:
- Title: `REQUEST SERVICE`
- Body: `"Request {service.name} from {expertName} for {price}. They'll review your goals before accepting."`
- **Required** textarea: *"What are you hoping to achieve?"* (max 400 chars; Send disabled until non-empty) — this is the `requestMessage` the expert reads on #22.
- Footer: `Cancel` + `Send request`

On `Send` → `requestService({userId, expertServiceId, message})` inserts a `pending` `ServiceRequest`, closes the modal, footer flips to `Request pending`.

### Review modal (after a completed engagement)
Triggered by `Leave a review`. A 1–5 **star picker** + a text area (max 400), `Cancel` / `Submit review` (disabled until rating ≥ 1 and body non-empty). On submit → `submitReview({expertUserId, userId, serviceRequestId, rating, body})` adds an `ExpertReview` (rendered on #6.1) + bumps the expert's `reviewCount`; footer flips to `✓ Reviewed`.

## Edges

- **From:**
  - Experts (#6) — Service Listings sub-tab tap
  - Expert Detail (#6.1) — Service listings row tap
- **To:**
  - Back (←) → `navigate(-1)` (returns to whichever surface opened this page)
  - Expert Detail (#6.1) — tap the "Offered by" mini card

## Data touched

- **Reads:** one `ExpertService` (by `ExpertServiceID` query param); the joined expert `User` + their `ExpertProfile` (mini card); the current user's `ServiceRequest` rows (the active, non-cancelled one drives the footer); that engagement's `Deliverable` rows (From-{expert} section); the user's `ExpertReview` for this engagement (reviewed vs not)
- **Writes:** `requestService({userId, expertServiceId, message})` on Send (inserts a `pending` `ServiceRequest`); `submitReview({...})` on Submit review (inserts an `ExpertReview` + bumps `ExpertProfile.reviewCount`)

## Notes / non-obvious

- **Request, not purchase.** There's no payment in the mock — the CTA sends a *request* with a required goal message; the expert accepts (`completed`) or declines (`cancelled`) on #22 Requests. "Completed" means *accepted*, not paid.
- **Pinned footer not inline CTA.** Same reason as #11.3 Challenge Detail's pinned Join button: the bullet list + expert card can push an inline button off-screen on Flutter's iPhone viewport. Footer guarantees the primary action is always reachable.
- **One review per engagement.** The `Leave a review` action only appears for a `completed` request that the client hasn't reviewed; submitting flips to `✓ Reviewed`. The written review shows on #6.1; `ExpertProfile.ratingAvg` stays the stored aggregate (a single review barely moves a large-N average), only `reviewCount` bumps.
- **No re-request flow in v1.** A single non-cancelled request drives the footer; declined (`cancelled`) requests don't count, so the client could request again after a decline.
- **Premium (future):** Premium-member discount, gift a request to a friend, schedule the kickoff call inline.
