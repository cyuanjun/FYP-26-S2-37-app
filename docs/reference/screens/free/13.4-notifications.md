# 13.4. Notifications

**Purpose:** Per-user notification preference management. Toggles all 12 notification types grouped into 4 categories. Reached from Profile (#13) → "Notifications" menu row.

## Layout

Vertical, scrollable. No bottom nav. Back arrow → Profile (#13). Same iOS grouped-settings pattern as #14 Account Settings.

## UI elements

### Top bar
- ← back arrow + "NOTIFICATIONS" in **Title 1**-ish (26 px `ink` font-black uppercase tracking-tight)

### Four toggle sections
Each section: `SectionLabel` + a column of `SettingRow`s with `Toggle` accessory. Toggles are **controlled** (read from / write to the store on every flip — no save button needed).

Each row shows:
- **Label** in **Body** (17 px `ink` font-semibold) — e.g. "Daily reminder"
- **Description** in **Subheadline** (15 px `muted`) — e.g. "Nudge me at my preferred time"
- **Toggle** on the right (lime when on, faint when off)

**Categories + types** (12 total):

- **Workout Reminders** — Daily reminder · Missed workout · Inactivity reminder
- **Summaries** — Weekly summary
- **Social** — Friend activity · Likes & comments · Challenge invites
- **Marketing & Updates** — Product tips · App updates · Promotions

**Premium-only types** (planned, not yet shown): *Rest day suggestion* (recovery-day prompts) will land here gated by `User.Role IN ('premium', 'expert')`. Gating mechanism TBD — likely a `RequiresPlan` field on the type metadata or a separate `PremiumNotificationPreference` table.

Display labels/descriptions live in `app/src/state/types.ts` (`NOTIFICATION_TYPES`) — UI strings, not DB columns.

Colours from [../../palette.md](../../palette.md), type sizes from [../../typography.md](../../typography.md). Reuses: `SectionLabel`, `SettingRow`, `Toggle`.

## Edges

- **From:** Profile (#13) — tap "Notifications" menu row
- **To:**
  - Back (←) → Profile (#13) — hard-wired `Link`

## Data touched

See [../../database-v1.md](../../database-v1.md).

- **Reads:** all `NotificationPreference` rows for the current user
- **Writes (instant on toggle flip):** `setNotificationEnabled(userId, typeKey, enabled)` — upserts the (UserID + TypeKey) row with new value + `UpdatedAt = now`

## Notes / non-obvious

- **No Save button.** Toggles are independent, no batch makes sense. Each flip commits immediately.
- **Per-(user, type) row** rather than one row per user with N boolean columns. Adding a new notification type later only needs a new enum value, not a schema migration. Tradeoff: more rows in the table.
- **FKs to User, not UserFitnessProfile.** Notifications apply to all roles (Admin gets product / app-update notifications too). Don't gate by athlete-only specialization.
- **All 12 types are seeded for the demo user** with sensible defaults — workout/progress/social on by default, marketing off (people don't love marketing).
- The `Toggle` component now supports controlled mode (`on` + `onChange` props) alongside its original uncontrolled mode (`defaultOn`). Existing usages (e.g. Login's "Remember me") keep working unchanged.
