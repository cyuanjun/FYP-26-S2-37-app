# 13.1. Fitness Profile (sub of #13)

**Purpose:** Captures the user's physical baseline + training context so the app can recommend workouts, estimate calorie burn, calibrate HR zones, and respect dietary / injury constraints when generating plans. Reached from Profile (#13) ŌåÆ "Fitness Profile" menu row.

**Sub-numbering convention:** `13.x` here means *"full screen reached from #13's menu"* ŌĆö distinct from `4.1`, which was a *state-overlay* of #4. Same `.x` suffix, different relationship type. Future Profile sub-pages (Fitness Goals 13.2, Submit Feedback 13.3) follow this pattern.

## Layout

Vertical, scrollable. **No bottom nav.** Content runs longer than the 780 px content area, so the user scrolls to reach Save Profile at the bottom ŌĆö explicit save makes sense here because multiple chip selections batch naturally.

## UI elements

### Top bar
- ŌåÉ back arrow + "FITNESS PROFILE" in **Title 1**-ish (26 px `ink` font-black uppercase tracking-tight)

### BODY METRICS section
Four `SettingRow`s, label above value, chevron-right, divided by `faint`:
- **Date of Birth** ŌĆö value composed as "12 Mar 2002 ┬Ę 24 yrs" (age computed from DOB at render time)
- **Sex** ŌĆö value "Female" / "Male" / "Other"
- **Height** ŌĆö value "168 cm" (or "5'6\"" when `User.PreferredUnits === 'imperial'`)
- **Weight** ŌĆö value "62 kg" (or "137 lbs" when imperial)

Tapping each row opens a per-field edit sub-page (TBD).

### ACTIVITY LEVEL section
Single full-width rounded-2xl `surface` card (not a `SettingRow` ŌĆö needs more emphasis because the value is descriptive, not just a number):
- *Title:* "Moderately Active" ŌĆö **Body** (17 px `ink` font-bold)
- *Subtitle:* "3-4 workouts per week" ŌĆö **Subheadline** (15 px `muted`)
- *Right:* chevron in `muted`
- Tappable; opens picker showing all 4 levels (Sedentary / Light / Moderate / Active) with their descriptions

### TRAINING EXPERIENCE section
Single-select chip row: **Beginner ┬Ę Intermediate ┬Ę Advanced**. One always selected (`Chip` with `selected={true}` fills `accent`; unselected are outlined `faint`).

### PREFERRED WORKOUTS section
Header: `Preferred Workouts` label on the left + small accent `+` icon button (18 px) on the right (right-aligned via `SectionLabel`'s `action` slot). Chip row below: **Running ┬Ę Strength ┬Ę Cycling** ŌĆö fixed default list, each chip togglable for quick inline edits. Tap the **`+`** in the header to open a `PickerModal` with the full catalog (Swimming, Yoga, HIIT, Pilates, Boxing, Climbing, etc.); selections persist in state regardless of whether the item appears as a chip on the main screen.

### DIET section
Same `+`-in-header pattern. Default chips: **Vegetarian ┬Ę Vegan ┬Ę Pescatarian**. Separated from Allergies because they're conceptually different (preference vs medical constraint).

### ALLERGIES section
Same pattern. Default chips: **Nuts ┬Ę Dairy ┬Ę Gluten**.

### INJURIES / LIMITATIONS section
Same pattern. Default chips: **Knee pain ┬Ę Lower back ┬Ę Shoulder**. Used to filter workout recommendations (e.g. hide high-impact runs for users with knee pain).

### SAVE PROFILE button
Full-width primary CTA at the bottom of the scrollable area. **Body** (17 px) font-black uppercase tracking-wider, `accent` bg, `bg` text, accent glow shadow. Commits all chip-section changes + any field edits in one transaction.

**Why explicit save here but not on #14 Account Settings?** Account Settings uses inline commit (per-field sub-pages, instant Segmented toggle). Fitness Profile has multiple multi-select chip sections that batch naturally ŌĆö committing on every chip tap would feel chatty. Save Profile groups the intent.

Colours from [../../palette.md](../../palette.md), type sizes from [../../typography.md](../../typography.md). Reuses: `SettingRow`, `SectionLabel`, `Chip` (new), inline ChipRow helper.

## Data touched

See [../../database-v1.md](../../database-v1.md).

- **Reads:** `User.DateOfBirth`, `User.Sex`, `User.HeightCm`, `User.WeightKg`, `User.ActivityLevel`, `User.TrainingExperience`, `User.PreferredUnits` (for unit display), plus the user's M:N selections from `UserWorkoutPreference`, `UserDietaryPreference`, `UserAllergy`, `UserInjury`
- **Writes (on Save):** updates the above User columns; replaces the user's rows in each junction table to match current chip selection. Single transaction.

## Notes / non-obvious

- **Height/weight stored in metric only.** `HeightCm` / `WeightKg` are the source of truth; conversion to imperial happens at display time based on `User.PreferredUnits`. Never store both.
- **Sex stored as biological enum** (`female` / `male` / `other`) because it's an input to BMR / calorie calculations, not an identity field. If we ever want a separate gender-identity field for social/profile contexts, that's a new column.
- **Many-to-many catalogs** (WorkoutType, DietaryPreference, Allergy, Injury) are seeded with a fixed initial catalog; users can pick from them but can't create custom entries in v1.
- **Overflow affordance lives in the section header**, not the chip row. Each chip-based section's `SectionLabel` takes an `action` slot ŌĆö here it's a small accent `+` icon button that opens the `PickerModal` for the full catalog. Always in a consistent position, never wraps with chips, never orphans on its own line. Header = action, chips = current state.
- **PickerModal is reusable** (`app/src/components/PickerModal.tsx`). Takes title + options + selected Set + onToggle + onClose. Train, Social, and any future multi-select picker uses this same component.
- **Search + add custom entries.** PickerModal has a search input at the top that both filters the catalog as you type and, when the query has no exact match, surfaces an `+ Add "X" as new` row at the top of results. Tapping it adds the custom string to your selection. The catalog tables (`WorkoutType` / `DietaryPreference` / `Allergy` / `Injury`) gain `IsCustom: boolean` + `CreatedByUserID: UUID nullable` columns to record user-added entries. Critical for Allergies and Injuries where no curated list will ever fully cover real-world specificity ("Sesame", "Sciatica", etc.). User-added entries are tagged "CUSTOM" with a small pill in the picker list, and remain selectable/deselectable like catalog entries.
- **List ordering in the picker:** *selected + custom* (top) > *selected* > *custom* > *default catalog* (bottom). Within each tier the original insertion order is preserved (stable sort). This surfaces the user's own picks and contributions first, while keeping the catalog scannable in its intended order at the bottom.
- **Chip row shows fixed defaults only.** Picker-only selections live in state (so Save Profile sends them) but don't surface as chips on the main screen ŌĆö to see/edit them, tap EDIT to reopen the picker. Simpler mental model than mixing visible and selected.
