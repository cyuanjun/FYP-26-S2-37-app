# 11.2. User Profile (sub)

**Purpose:** The public profile page for any other user (or a self-view that's identical minus the Friend toggle). Surfaces identity (avatar / name / handle), the friendship state with the current user, and a chronological index of every `Post` they've authored. Reached by tapping a user's name or avatar anywhere in the Social surfaces (#11 feed, #11.1 Post Detail, #11 Friends modal, #11 search results).

Earlier iterations didn't have this — every other-user reference on Social was a dead text node. Once posts went polymorphic and the friend graph became real, the absence started to feel like a hole: you could see who liked / commented / placed in a competition but never go check them out. #11.2 plugs that hole with the smallest surface that resolves it.

## Layout

Vertical, scrollable. **No bottom nav** (sub-page). Back arrow → Social (#11). Identity is centered top-down; everything else (stats / Follow / About / Posts) stacks beneath in left-aligned cards.

## UI elements

### Top bar
- Left: ← back arrow + the target's `@username` in Display 22 px font-black uppercase (the *username* is the page title — no static "PROFILE" label).
- Right: a `⋯` more-options icon button (stub in v1 — future hook for share-profile / report / block).

### Identity block (centered)
- 128 px circular avatar on `surface-2` background — initials fallback in `accent` font-black 44 px (first letter of first + last name, e.g. `MP`)
- Full name below in Display 22 px font-black uppercase, `ink`
- Meta line in Caption 2 muted: `@handle · {tagline} · {city}`, segments joined by ` · `. Tagline and city are optional — only present segments render, so a user without a tagline shows just `@handle · Singapore`, and a user with neither shows just `@handle`.

### Stats tile
A single `surface` card with `ring-1 ring-faint/60`, three equal columns separated by `divide-faint/40` vertical dividers:
- **Workouts** — count of `WorkoutSession` rows for this user where `EndedAt IS NOT NULL`, all-time (bypasses the Free monthly history cap on purpose — same rationale as #13 Profile's Workouts tile).
- **Friends** — count of `Follow` rows where `followerId === target.userId` (= the count of people *they* follow, which under the mutual model is identical to their follower count).
- **Active days** — count of distinct calendar days (`endedAt.slice(0, 10)`) across the same ended-sessions set, all-time.

Each column: large 26 px font-black number on top, Caption-2 muted uppercase label underneath.

**Why three columns and not Followers / Following.** The mockup borrowed the Instagram/Twitter triplet (Posts / Followers / Following), but Wise Workout's follow graph is **mutual** (see #11 "Mutual friendship"), so Followers and Following would always be identical — showing both would be visually heavy and dishonest. The substitution is Workouts + Friends + Active days, all derivable from real data and signalling what a fitness app actually cares about (activity, social reach, consistency).

### Friend toggle
A full-width 14 px font-black uppercase pill below the stats tile, **hidden when viewing self**. Label flips with state and is **action-first** (what tapping does), not state-first (what is currently true):
- `Add Friend` → filled `accent` CTA, py-4 rounded-2xl
- `Unfriend` → outlined `bg` chip with `ring-1 ring-faint`, muted text; `danger` hover warns about the destructive action

Backed by the same `followUser` / `unfollowUser` actions as the Social Friends modal and search results — mutual by design (see #11's "Mutual friendship" section). Tapping is instant, no confirmation.

### About section (conditional)
Renders only when `User.bio` is non-null. Header `ABOUT` in Caption-2 muted uppercase, then the bio paragraph inside a `surface` card with `ring-1 ring-faint/60`, 14 px ink text. Users without a bio simply don't get this section — no placeholder.

### Recent Posts section
- Section header: `Recent Posts · N` (muted, Caption 2, uppercase tracking) — when `N === 0`, header reads just `Recent Posts`.
- One compact link card per post, newest first. Each card has:
  - Post-kind icon (🏋️ for `workout_share`, 🏆 for `challenge_result`)
  - Tiny `WORKOUT SHARE` / `COMPETITION RESULT` eyebrow in muted Caption 2
  - Post title — workout's `customName ?? workoutType.name` for shares, `challenge.name` for results
  - Relative time below the title
  - Optional caption (`Post.body`) — clamped to two lines
  - Footer strip with `❤ N` + `💬 N` counts (read-only here — interactions happen on Post Detail)
- The whole card is a `Link` to **#11.1 Post Detail** for that post

Empty state: muted text — `"You haven't shared any posts yet."` for self, `"{firstName} hasn't shared any posts yet."` for other users.

**Why compact cards, not the full feed cards.** The full Social-feed cards are the canonical interactive surface (heart toggle, comment chip, embedded WorkoutListCard). Duplicating them on the profile would make the profile page essentially "a feed scoped to one user", which is fine but heavy. The compact cards are an *index* — tap any one to drop into the real interactive surface (#11.1). Trades a click for a much lighter page and avoids re-implementing the full card plumbing here.

## Entry points (name-tap affordances)

Every name + avatar across the Social surfaces routes here. The implementation is consistent: avatar + name area is a `Link` to `/free/11.2-user-profile?id=<userId>`; sibling action buttons (Friend toggle, heart, etc.) use `stopPropagation` so they don't compete with the link.

| Surface | What's tappable |
|---|---|
| **#11 Social** — feed post AuthorRow (workout_share + challenge_result) | avatar + name + `@handle · time` |
| **#11 Social** — challenge_result ranking rows | each participant's name (medal column + metric column unaffected) |
| **#11 Social** — search results UserRow | avatar + name area (Friend toggle stays a button) |
| **#11 Social** — Friends modal UserRow | avatar + name area (Friend toggle stays a button) |
| **#11.1 Post Detail** — author row | avatar + name + `@handle · time` |
| **#11.1 Post Detail** — challenge_result ranking rows | each participant's name |
| **#11.1 Post Detail** — comment cards | avatar + author name |

The post card on #11 is itself a `Link` to Post Detail, so inner name-Links call `e.stopPropagation()` to win the tap.

## Self-view

When `userId === current.userId`, the page renders identically except:
- The Friend toggle is hidden (self-follow makes no sense)
- The empty-state copy changes to second person (`"You haven't shared any posts yet."`)

The bio / tagline / city render exactly as a non-owner would see them — this is a deliberate "preview your public face" affordance, important since the same fields are editable elsewhere (eventually on #13 Profile's edit flow). Editing those fields lives on #13, not here.

We don't redirect to **#13 Profile** for self-view because the two pages answer different questions. #13 is the *current user's identity hub* — settings, fitness profile, badges, lifetime stats, all-time workouts. #11.2 is the *public social face* — what other people see when they tap your name. Keeping them separate means #13 can hold settings without leaking them to #11.2 viewers.

## Edges

- **From:** any name-tap affordance listed above (Social, Post Detail, Friends modal, search results, comments)
- **To:**
  - Back (←) → Social (#11)
  - Post Detail (#11.1) — tap any post card

## Data touched

See [../../database-v1.md](../../database-v1.md).

- **Reads:** the target `User` (by `UserID` query param — could be current user or someone in `otherUsers`); `User.bio`, `User.tagline`, `User.city` for identity + About; all `WorkoutSession` rows scoped to `target.userId` with `EndedAt IS NOT NULL` (for Workouts + Active-days counts); all `Follow` rows (for `isFriend` + Friends count); all `Post` rows authored by the target (sorted by `CreatedAt DESC`); per-post `PostLike` + `PostComment` counts; for `workout_share` posts the linked `WorkoutSession` + `WorkoutType` (for title); for `challenge_result` posts the linked `Challenge` (for title)
- **Writes:**
  - `followUser(currentUserId, targetUserId)` — Add Friend tap
  - `unfollowUser(currentUserId, targetUserId)` — unfriend tap (when current state is `Friends`)

## Notes / non-obvious

- **Stats are derived, never stored.** Workouts / Friends / Active-days are computed from `WorkoutSession` + `Follow` rows on every render. No `User.workoutsCount` cache to keep in sync.
- **Mutual follow, single Friends count.** A directional Followers / Following split would have to fake the asymmetry — we deliberately collapse both into one Friends column instead. See "Why three columns" above.
- **Bio / tagline / city are nullable.** A user with none of them shows just `@handle` as the meta line and no About section — page still feels complete (avatar + name + stats + posts carry it).
- **`⋯` menu is a stub.** Button renders for the right-side balance and the conventional affordance, but has no onClick. Wire up share-profile / report / block when those features get specced.
- **Compact post cards, not full feed cards.** Tapping a card routes to #11.1 where the full interactive surface lives. Keeps #11.2 a lightweight index rather than a second feed.
- **Numbering note.** The old #11.2 slot held "Compose Post" which was deleted when workouts-are-posts shifted; the slot is reused here. Per [../../screens-v1.md](../../screens-v1.md) decimal numbering, this is a single-parent sub of #11 Social.
- **Premium (future):** badges grid, cover photo, streak counter as a 4th stat, "you were both at the same challenge" hints, follower-graph visualisation (since friends are mutual, this is mostly a UI flourish).
