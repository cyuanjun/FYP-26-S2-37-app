# 6.1. Expert Detail (sub)

**Purpose:** The full profile page for one expert — identity, stats, about, credentials, specialties, and their service listings. Reached only from #6 Experts (tap any ExpertCard) or from #6.2 Service Detail's "Offered by" mini card.

## Layout

Vertical, scrollable. **No bottom nav** (sub-page). Back arrow → `navigate(-1)` so the user returns to the sub-tab + scroll position they came from on #6 (same back-pattern as #11.3 Challenge Detail).

## UI elements

### Top bar
- Left: ← back arrow
- Right: **circular Follow toggle** (heart). Filled `accent` when followed; outlined `faint` ring when not. Backed by `toggleFollowExpert(currentUserId, expertUserId)` — instant flip, no confirmation.

### Identity block (centred)
- 112 px circular avatar (initials fallback, `accent` glyph on `surface-2`)
- Caption-2 muted: `— {profile.title}` (e.g. *— STRENGTH COACH*)
- Display 22 px font-black uppercase name
- Accent 13 px line: `{yearsCoaching} years coaching {first specialty}`

### Stats tile
Three columns in a `surface` card, divided by `divide-faint/40`:
- **Rating** — `profile.ratingAvg.toFixed(1)` (e.g. `4.9`)
- **Reviews** — `profile.reviewCount`
- **Clients** — `profile.clientCount`

All three are stored directly on `ExpertProfile` for the mock; in production they'd be computed (rating from `ExpertReview`, clients from purchase aggregation).

### About
Section header `ABOUT` (muted Caption 2 uppercase) + `profile.about` paragraph in ink 14 px.

### Credentials & experience
Section header + a `surface` card. Each `profile.credentials[]` entry renders as a row: accent check glyph + the credential string.

### Specialties
Section header + a row of filled-accent pills, one per `profile.specialties[]` entry.

### Service listings
Section header + count caption + a list of compact `surface` cards. Each card: service name + 1-line description + price (right-aligned in display) + `View ›` accent caption. Tap → #6.2 Service Detail. Empty state: muted *"No services listed yet."*

## Edges

- **From:** Experts (#6) — tap an ExpertCard; Service Detail (#6.2) — tap the "Offered by" mini card
- **To:**
  - Back (←) → `navigate(-1)` (preserves the sub-tab + scroll on #6 the user came from)
  - Service Detail (#6.2) — tap any service row

## Data touched

- **Reads:** the target `User` (by `UserID` query param) + their `ExpertProfile` (shared-key join); all `ExpertService` rows for `expertUserId = target.userId` (sorted by `priceCents` ascending for the listings section); `User.followedExpertIds` for the current user (to compute `isFollowed`)
- **Writes:** `toggleFollowExpert(currentUserId, expertUserId)` — heart-toggle in the top bar

## Notes / non-obvious

- **Heart-toggle is the only follow mechanism.** No "Follow" text button — the icon is the entire UX. Matches action-first labeling principles: the toggle's visual state IS the action ("currently following" / "tap to follow").
- **`navigate(-1)` back.** Like #11.3 Challenge Detail; preserves the Experts/Services sub-tab + search/category state on #6 instead of resetting it.
- **No "Following" section here yet.** The follow toggle commits to `User.followedExpertIds`; a dedicated "Following" view (likely a sub-tab on #6 or section on #13 Profile) is future work — see #6 spec.
- **Distinct from the user-to-user `Follow` entity in Social** (which is mutual friendship). Expert "follow" is a one-way bookmark of marketplace experts, materialised as a list on `User` rather than a junction.
- **Premium (future):** see review-body text (Free sees rating + count only), book a free 15-min intro call, see "verified" badge on credentials.
